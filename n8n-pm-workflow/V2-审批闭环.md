# V2：多维表格配置源 + 飞书审批闭环

`workflow-v2.json` 在 V1 基础上做了两处升级。

---

## 升级一：配置源改为真实的飞书多维表格

V1 用 Code 节点内置 `CONFIG` 映射；V2 改成在线读飞书多维表格，PM 可自助维护。

新增 / 替换的节点链：

```
解析需求字段
  → 获取飞书Token (换 tenant_access_token)
  → 查询项目配置(多维表格) (按 project_id 过滤)
  → 提取配置字段 (Code，拍平富文本 → 顶层字段)
  → 需求评估 Agent ...
```

### 飞书 API 说明

1. **获取 token**：`POST /open-apis/auth/v3/tenant_access_token/internal`，
   body 填 `app_id` / `app_secret`，返回 `tenant_access_token`（有效期 2h）。
2. **查询记录**：`POST /open-apis/bitable/v1/apps/{app_token}/tables/{table_id}/records/search`，
   header 带 `Authorization: Bearer {token}`，body 用 `filter` 按 `project_id` 精确匹配。
3. **提取字段**：飞书文本字段返回的是富文本分段数组，`提取配置字段` 节点的
   `flat()` 函数把它们拍平成纯字符串，再挂到顶层供后续节点引用。

### 需替换的占位符

| 占位符 | 替换为 |
|--------|--------|
| `REPLACE_APP_ID` / `REPLACE_APP_SECRET` | 飞书自建应用的凭证 |
| `REPLACE_APP_TOKEN` | 多维表格的 app_token（表格 URL 里那段） |
| `REPLACE_TABLE_ID` | 表的 table_id |

> **安全建议**：生产环境别把 app_secret 硬编码进 jsonBody。改用 n8n 的
> **Header Auth / 通用凭据**，或单独做一个「取 token」子工作流并缓存（避免每次请求都换 token）。

### 配置表字段（飞书多维表格列名要一致）

`project_id`、`project_name`、`eval_system`、`effort_system`、`model`、
`pass_threshold`、`effort_baseline`、`feishu_webhook`、`gitlab_project`
（含义见 `SOLUTION.md` 7.2 节）。

---

## 升级二：飞书审批闭环

V1 的工时卡片只是通知；V2 让**项管在飞书点按钮确认/驳回**，n8n 挂起等待，
按结果分流回写。

新增节点链：

```
工时评估 Agent
  → 飞书-工时审批卡片 (含两个按钮，URL 指向 n8n 恢复地址)
  → 等待项管审批 (Wait, 模式=On Webhook Call)   ← 流程在此挂起
  → 审批是否通过 (IF, query.action == 'approve')
       ├─ 通过 → GitLab 回写-确认
       └─ 驳回 → 飞书通知-审批驳回(让 PM 重评)
```

### 机制原理（关键）

- `等待项管审批` 节点设为 **`resume: webhook`**，n8n 会为本次执行生成一个
  一次性恢复地址，表达式 `{{ $execution.resumeUrl }}`。
- 审批卡片的两个按钮用 `url` 动作，分别指向：
  - 确认：`{{ $execution.resumeUrl }}?action=approve`
  - 驳回：`{{ $execution.resumeUrl }}?action=reject`
- 项管点按钮 → 浏览器请求该地址 → **唤醒挂起的执行**，并把 `?action=` 带进来。
- `审批是否通过` 节点读 `{{ $json.query.action }}` 判断走哪条分支。

> **为什么用这个方案而不是飞书审批 API？**
> 飞书审批实例的状态回调要走「应用事件订阅」，需要再建一条回调工作流做
> `审批实例 → resumeUrl` 的映射，复杂且要存状态。用「卡片按钮直连 resumeUrl」
> 自包含、零额外存储、最易跑通。代价：谁拿到链接谁能点（适合内网可信环境）。
> 若需严格鉴权 / 留痕，再升级为飞书审批 API + 事件订阅。

### 注意事项

- `$execution.resumeUrl` 在 Wait 节点**之前**的卡片节点里即可引用（n8n 会预生成）。
  导入后若该表达式为空，把卡片节点移到 Wait 之后并改用「先 Wait 拿 URL 再发卡」的
  顺序，或在 Wait 节点开启「Limit wait time」兜底防止永久挂起。
- 恢复地址默认 GET 也能触发；按钮点击是浏览器跳转，会打开一个空白页——
  可在 Wait 节点配置 `responseMode` 返回一段「已收到，可关闭页面」的提示。
- Wait 挂起期间执行不占资源，但**建议设最长等待时间**（如 3 天），超时自动按驳回处理。

## 导入与切换

1. n8n 导入 `workflow-v2.json`。
2. 替换占位符：飞书 app 凭证、多维表格 token、LLM 凭据、GitLab 凭据。
3. 飞书多维表格建好配置表并填至少一行（含一个真实 `project_id`）。
4. 先用 `curl`（见 `SOLUTION.md` 9.1）打 Webhook，走通到工时卡片。
5. 点卡片按钮，确认执行被唤醒并正确分流。

> V1（`workflow.json`）适合快速演示 / 离线跑；V2 适合接真实飞书环境与人工审批。
> 两者可共存，按需启用其一。

## ⚠️ 说明

`workflow-v2.json` 为结构正确的骨架，未在真实 n8n 实例实跑。除通用注意事项外，
本版额外依赖：飞书应用权限（`bitable:app`、`im:message`）、多维表格列名与代码一致、
`$execution.resumeUrl` 在你的 n8n 版本可用。导入后请逐节点验证 Input/Output。
