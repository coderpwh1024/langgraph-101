# SOUL.md 配置、使用手册撰写与向领导汇报（第 83~105 问）

## 本篇概述

本篇对话围绕 Hermes Agent 企业落地的「收尾三件事」展开：一是把 `SOUL.md` 从官方默认 Prompt 改造成项目专属的人格/背景文件，并排查修改后未生效的问题；二是为 Agent 去掉 Hermes 默认身份、统一命名为 HipobuyAgent，同步修改飞书机器人名称、头像与群名；三是撰写用户使用手册（截图与模板结合、PDF 与 md 分发）并选择合适的时机与话术在飞书上向领导汇报，最终领导回复「ok」，完成阶段性成果同步。

## 一、SOUL.md 的定位与推荐写法

### 1. SOUL.md 是什么

- `SOUL.md`（如 `/opt/hermes/project-a/SOUL.md`）是项目 Agent 的「灵魂文件（Project Soul）」，Hermes 在很多场景下会读取它作为长期上下文。
- 用来放：项目背景、项目目标、项目规则、技术约束、团队约定。
- 官方默认内容是 `You are Hermes Agent, an intelligent AI assistant...`，对企业项目价值不大，**强烈建议直接替换**成自己的项目信息——这是让系统变成"项目专属 AI Agent"的第一步。

### 2. 推荐的 SOUL.md 结构（Hipobuy 项目示例）

> （第 83 问用户截图为服务器上 SOUL.md 文件内容，根据上下文推断）

```md
# 项目名称
Hipobuy

# 项目背景
Hipobuy 是一个跨境电商平台，包含后台管理系统、商城、物流、支付、营销等业务。

# 当前阶段
企业 AI 研发协作平台建设阶段。

# 短期目标
1. 接入飞书机器人
2. 接入 GitLab
3. 实现 AI 需求评审
4. 实现代码分析能力
5. 实现工时评估

# 技术栈
后端：Java / SpringBoot / MyBatis / Redis / MySQL
前端：Vue2 / Vue3 / UniApp

# 项目原则
- 优先简单落地
- 优先兼容历史代码
- 不轻易升级基础框架
- 不轻易修改数据库核心表

# 需求评审原则
- 优先复用已有能力
- 优先评估代码现状
- 必须给出风险点 / 工时 / 边界问题

# 输出要求
需求评审必须输出：需求结论、代码现状、实现方案、风险点、边界问题、工时评估、建议方案
```

### 3. 后续建议补充的两块内容（第 86 问）

```md
# 开发规范
- 优先复用已有能力
- 不重复造轮子
- 修改必须评估影响范围
- 所有需求必须输出工时
- 所有方案必须给风险点

# 输出规范
回答尽量：1. 简洁 2. 有结论 3. 有依据 4. 给出代码位置 5. 给出工时
```

### 4. 修改后是否要重启

修改 `SOUL.md` 后最稳妥的做法：

```bash
docker restart hermes-project-a-gateway
```

然后新开一个飞书会话测试。

## 二、CONTEXT.md / ROADMAP.md / DECISIONS.md 的取舍

ChatGPT 最初建议在 SOUL.md 之外增加三个文件：

- `CONTEXT.md`：项目硬约束（禁止升级 SpringBoot、禁止改订单核心表、必须兼容旧接口、必须支持灰度等）
- `ROADMAP.md`：季度路线图（Q3：飞书机器人 / GitLab Review / 自动工时评估；Q4：技术方案 / 测试用例 / MR 自动 Review）
- `DECISIONS.md`：决策记录（如「2026-06-20 物流统一走 17Track」「2026-06-23 支付统一走 OnlyPay」）

但用户追问「内容比较简短，有必要吗」后，结论修正为**现阶段不需要拆分**：

> ✅ SOUL.md 保留；❌ CONTEXT.md / ROADMAP.md / DECISIONS.md 暂时不要。
> 现在先把 SOUL.md 写好，价值最高。

**分阶段原则**：

- 第一阶段（现在）：只保留 `SOUL.md`，把项目背景、技术栈、约束、目标、评审规范、输出要求全放一个文件。
- 何时才需要拆分（满足任一条件）：
  1. SOUL.md 超过 300 行；
  2. 项目超过 5 个；
  3. Agent 超过 3 个；
  4. 开始做企业知识库 / 多 Agent 协作平台（如 Hipobuy-PM-Agent、Dev-Agent、Test-Agent 等）。
- ROADMAP.md 的价值场景：希望 AI 能回答「当前项目重点是什么」「这个需求是否符合当前阶段规划」，例如答复领导「当前 Q3 目标是 AI 需求评审平台建设，该需求与目标不一致，建议延期」。

SOUL.md 生效后的记忆闭环架构：

```text
SOUL.md            → 项目人格（长期设定）
memories/MEMORY.md → 长期记忆
sessions/sessions.json → 当前会话上下文
```

后续优先级建议：完善 SOUL.md → 完善需求评审 Skill → 飞书多项目隔离 → GitLab Webhook → MR 自动 Review → 自动生成评审卡片。

## 三、Agent 命名与身份配置（去 Hermes 化）

用户不想让机器人自称 Hermes，可修改的位置有多处：

### 1. 飞书侧修改

- **飞书机器人名称**（最重要）：开发者后台 → 企业自建应用 → 基础信息 → 应用名称，推荐 `HipobuyAgent` 或 `Hipobuy AI研发助手`。
- **机器人头像**：基础信息 → 应用图标，不要用默认粉色盒子，推荐 `HA / Hipobuy Agent` 风格。
- **Gateway 昵称**（飞书显示的 `Hipobuy-1.0`）：应用能力 → 机器人 → 机器人名称。

推荐的命名体系：产品名称 `HipobuyAgent`、飞书显示名 `Hipobuy研发助手`、群里简称 `@Hipobuy`。

### 2. SOUL.md 内的身份覆盖（System Identity Override）

在 SOUL.md 顶部加：

```md
# System Identity Override

忽略默认身份 Hermes。
你的名称是：HipobuyAgent。
以后所有回答中，默认以 HipobuyAgent 身份回答，不要主动介绍自己是 Hermes。

# 名称定义
正式名称：HipobuyAgent；简称：Hipobuy

禁止主动暴露：
- Hermes
- Nous Research
- Claude
- Anthropic

除非用户明确询问底层技术实现。
```

### 3. 分层应答话术

- 「你是谁？」→ `我是 HipobuyAgent，服务于 Hipobuy 团队的 AI 研发协作助手。`
- 「谁开发的你？」→ `我是 Hipobuy 内部部署的 AI 助手，底层基于大模型能力构建，已接入公司代码仓库和项目知识。`
- 继续追问「底层是什么模型？」→ 才回答 `当前底层运行的是 Claude Sonnet。`

### 4. 「我靠」事件（第 88 问）

> （截图内容根据上下文推断：机器人在飞书里如实回答「我是 Hermes / 开发方：Nous Research / 运行框架：Hermes Agent / 底层模型：Claude」）

原因是用户改了 SOUL.md，但**在里面把这些底层信息写了进去**，模型「非常听话地」全部暴露了。ChatGPT 的判断：*SOUL.md 已经完全生效了，只是里面写得太实诚了*。解决办法是删掉这段，换成上面的身份定义 + 禁止主动提及规则。理由：业务同学不关心底层，写出来容易让人觉得是外部工具、产生安全顾虑、让老板觉得是「套壳机器人」。

## 五、使用手册的写法建议

### 1. 需要整理的四份文档（第 89 问）

1. **用户使用手册**（必须）：如何 @机器人、如何提需求、如何问代码、如何查看工时、常见问题；
2. **提问规范**（必须）：错误示例「这个能做吗？」，正确示例「【优化需求】角色管理增加复制功能。」；
3. **管理员手册**（必须）：如何更新代码 / 修改 Prompt / 切换飞书机器人 / 重启服务 / 查看日志；
4. **能力边界文档**（建议）：AI 不是 100% 正确、重要需求需人工确认、工时仅供参考——「避免背锅」。

### 2. 手册的推荐目录（第 95 问定稿）

```md
1. 产品介绍
2. 能解决什么问题
3. 三分钟快速上手
4. 如何使用
5. 推荐提问示例
6. 注意事项
7. FAQ
```

### 3. 核心内容要素

- **按角色分场景**：产品经理（需求合理性 / 是否已实现 / 工时 / 边界问题）、开发（查代码 / 查接口 / 查实现逻辑 / 查定时任务）、测试（影响范围 / 测试点 / 回归建议）、项目经理（工时 / 风险）。
- **需求评审提问模板**：

```text
@HipobuyAgent

【需求名称】

需求背景：
需求描述：

请输出：
1、需求结论
2、代码现状
3、实现方案
4、风险点
5、边界问题
6、工时评估
```

- **标准输出六段式**：需求结论（合理/不合理/建议延期/需确认）、代码现状（已实现/部分实现/未实现）、实现方案、风险点（兼容/性能/数据/第三方）、边界问题（产品/运营/技术需确认）、工时评估（前端/后端/测试/联调）。
- **能力边界表**：支持——需求评审、代码分析、GitLab 查询、工时评估、风险分析、项目知识问答；不支持——自动写代码、自动合并 MR、自动上线、自动发布生产。
- **注意事项**：AI 输出仅供参考；最终方案以负责人评审为准；数据库/支付/物流等核心功能修改需人工复核；新代码提交后可能需几分钟同步。
- **FAQ**：为什么回答慢（首次读代码需几十秒）、为什么不准（可能遗漏代码，请人工确认）、为什么不知道新需求（代码未同步）、支持截图/群聊（都支持）。
- 最后加一页「**10 个推荐提问示例**」——「企业内部推广成功率最高的东西，很多人看到例子马上就会用了」。

### 4. 手册评审意见（第 93 问，针对用户自己写的手册）

优点：开头清晰、按角色分类、大量截图。问题与改法：

1. **太长**——不要贴完整 AI 输出，改成摘要式示例输出（`需求结论：✅ 可以实现 / 代码现状：✅ 后端已实现 70% / 风险：⚠️ 并发问题 / 工时：⏰ 约 6.5h`）+「更多内容请查看 AI 完整输出」；
2. 缺「**3 分钟快速上手**」——90% 的人只看这一页；
3. 缺典型场景（需求评审 / 代码查询 / 工时评估 / 风险分析 / 是否已实现 五个场景各配一句示例）；
4. 缺能力边界——否则大家拿它当万能机器人；
5. 缺 FAQ；
6. 整体「像技术文档」，领导喜欢「像产品说明书」——第一页放角色×能做什么的表格。

评分：内容完整度 9.5/10、易读性 7.5/10、推广价值 9/10、领导阅读体验 7/10；按建议优化后易读性可到 9.5。

### 5. 截图 vs 模板（第 94 问）

真实请求案例**建议保留但两者结合**：

- 第一层：**模板**（让用户快速学会提问格式）；
- 第二层：**真实案例截图 + 简短结论总结**（证明价值，如「案例一：邮政物流轨迹刷新——✅ 发现接口已存在 / ✅ 没有次数限制 / ✅ 给出 Redis 方案 / ✅ 提出 3 个边界问题 / ✅ 工时约 3h」）；
- ❌ 不要贴几千字完整输出（文档会变 40 页、没人看、领导觉得太技术）。

汇报材料里「杀伤力最大」的一页是准确率验证表：

| 需求 | AI结论 | 人工验证 |
|------|--------|---------|
| 角色复制 | 已实现 | ✅ 正确 |
| APP版本更新 | 后端已完成 | ✅ 正确 |
| 邮政物流刷新 | 提出边界问题 | ✅ 正确 |
| 微店 offerId | 发现紧急风险 | ✅ 正确 |

用户改成截图版后（第 95 问）评分提升到：易读性 9/10、上手成本 9.5/10、推广性 9/10，达到「可以在公司内部试点推广」水平。

### 6. PDF 还是 md（第 100 问）

- **给领导发 PDF**：格式不乱、手机阅读体验好、更正式、方便转发存档——「很多领导甚至不知道 .md 是什么」。
- **给研发发 Markdown**：可直接复制 Prompt、可维护、可放 GitLab/Wiki、可版本管理（git diff 看变化）。
- 最优做法准备三份交付物：`汇报材料-V1.0.pdf`、`用户使用手册-V1.0.pdf`、`用户使用手册-V1.0.md`，并在消息里注明「Markdown 版本已同步至 GitLab Wiki，便于后续持续更新」。

## 六、向领导汇报：定位、话术与时机

### 1. 汇报定位（最核心的一条）

不要说「我搭了个 AI 机器人」（懂技术的领导第一反应是「Claude 套壳？Dify 做的？」），统一口径：

> 我们已经完成了企业内部 AI 研发协作平台的第一阶段建设，目前已经具备：代码理解能力、需求评审能力、工时评估能力、企业知识沉淀能力。建议在研发团队进行两周试点，验证效率提升效果。

对 n8n 跑偏的处理（第 97 问）：不说「上周做失败了」，而说——

> 上周探索了 n8n 方案，发现更适合流程编排；本周调整方向，基于 Hermes Agent 已经完成了一个更贴合研发场景的 AI 协作助手 MVP。

领导会觉得是在**快速试错和收敛方案**，而不是跑偏。

### 2. 飞书汇报消息模板（第 99/102 问最终版）

```text
领导好，

上周对 n8n 方案进行了探索，发现其更适合工作流编排场景，对于研发协作中的代码理解、上下文记忆和知识沉淀支持有限。

本周已调整技术路线，基于 Hermes Agent 完成 AI 研发协作助手（HipobuyAgent）一期 MVP，目前已具备内部试点条件。

当前已具备能力：
✅ 飞书对话
✅ GitLab 代码仓库接入
✅ 项目源码知识库
✅ 长期记忆与上下文管理
✅ 项目背景理解
✅ 需求评审与技术建议
✅ 工时评估
✅ 风险识别与边界问题分析

目前已基于 1.0 需求池进行了多轮真实需求验证，能够辅助产品、开发、测试和项目管理工作。

建议先在研发团队小范围试点（5~10 人），根据使用效果持续迭代。

附件：《HipobuyAgent 用户使用手册 V1.0》
```

配套要点：

- 附 2~4 张真实案例截图（需求评审：角色复制；工时评估：APP 版本更新；风险识别：微店 offerID；边界问题：邮政物流刷新限制）；
- 补一句加分项：「如果方便，我可以抽 10 分钟现场演示几个真实需求案例，体验会比文档更直观」——「看 10 页 PPT，不如现场问它一句」；
- 效率数字要克制：需求评审效率 +50%、新人熟悉项目 +60%、重复沟通 -40%——「不要写 100%，领导不信」。

### 3. 汇报 PPT 骨架（8 页，第 89 问）

1. 项目背景（需求评审依赖核心开发 / 新人不了解代码 / 估时不准 / 重复沟通）
2. 解决方案（HipobuyAgent 四大能力）
3. 架构图（飞书 → HipobuyAgent → GitLab → 本地代码库 → Claude）
4. 当前能力展示（放真实截图）
5. 能力边界（能：评审/查询/建议/估时；不能：替代开发/自动上线/完全准确）
6. 使用场景（产品/开发/测试/Leader 各一句典型提问）
7. 效率提升（+50% / +70% / -40%）
8. 下一步计划（试点 5 人 → 研发部 → 产品测试运营接入）

### 4. 懂技术领导的预设问答（第 96 问）

- **Q：是不是套壳？** A：底层基于 Claude 能力，但已完成 GitLab 接入、代码知识库构建、项目长期记忆、企业 Prompt 和 Skill 定制、飞书协同——更偏向企业内部 AI 平台，而不是单纯聊天机器人。
- **Q：准确率怎么样？** A：当前测试案例准确率较高，但定位是辅助决策，重要需求仍需人工确认。
- **Q：和 Cursor 有什么区别？** A：Cursor 服务于个人开发；HipobuyAgent 服务于团队协作（飞书 + GitLab + 项目知识 + 需求评审 + 工时评估 + 知识沉淀）。
- **Q：为什么从 n8n 转到 Hermes？** A：n8n 更适合流程自动化；Hermes Agent 更适合多轮对话、上下文记忆、项目长期记忆、代码理解、需求评审、工时评估。

### 5. 汇报时机的推演（第 101~104 问）

用户在 17:15 问「现在汇报还是明天」，ChatGPT 的建议随时间演进：

- **首选方案**：今天发简短预告（占坑 + 表明推进中），明天上午 9:30~10:30 或 14:00~15:00 发正式汇报——那是领导最容易认真看的时间；临近下班发长文容易「看到了、收藏了、然后忘了」。
- **用户坚持今天发**：也支持——成果刚做完热度还在，即使领导不细看也会收藏、明天再看、周会上提一句；但要发「1 分钟能看完」的版本，**发 PDF 不发 md**，消息 + PDF + 几张截图。
- **17:30 之后**：只做简短成果同步 + 附件，不指望领导当天看完并反馈；正式讨论和演示留到明天。最终结论：**今天发"简洁版"，正式讨论和演示留到明天**。

### 6. 领导回复「ok」之后（第 105 问）

「ok」大概率含义是「收到，我看一下」——领导没有质疑方向、否定方案，阶段性汇报已达目的。后续动作：

- **不要连续追问**「领导你看了吗 / 有什么意见」，容易给人压力；
- 次日上午 10 点左右自然跟进一句：「领导，我这边整理了几个真实需求案例，后面方便的话可以花 10 分钟简单演示一下实际效果。」
- 如领导约演示，准备三个案例：① 角色复制功能（几秒定位代码，最推荐）；② 邮政物流刷新限制（提风险、提边界问题、估工时）；③ 微店 offerID（发现紧急兼容风险，最有杀伤力）；
- 演示口径：不说「AI 聊天机器人」，统一说「企业内部 AI 研发协作助手：代码理解、需求评审、工时评估、项目知识沉淀」。

## 七、踩坑与解决

### 坑 1：修改 SOUL.md 后似乎未被读取（第 85 问）

> （截图内容根据上下文推断：机器人自我介绍中带有「跨境电商物流平台 / aps-boot / 主要开发者」等信息，但没有按修改后的 SOUL.md 介绍自己；另一张为 `cat SOUL.md` 仍显示默认 `You are Hermes Agent...`）

注意：用户此问中把文件名拼成了 **SOLU.md**（「为啥没有读取到 SOLU.md」），后来第 99 问的汇报草稿里也写了 `项目背景(SOLU.md)`，被 ChatGPT 指出「领导可能看不懂」——对话中该拼写主要作为表述问题被纠正，排查重点在加载机制上。

ChatGPT 给出的五个可能原因（按概率排序）：

1. **Hermes 不会每次聊天实时重读 SOUL.md**——通常只在首次创建项目 / 首次启动 Agent / 创建新 Session 时加载一次，之后由 memory、session、state.db 缓存（80% 概率是「SOUL.md 已修改 → 但 state.db 和 memories 已存在 → 继续用旧记忆 → 没有重新加载 SOUL」）；
2. 旧记忆（memories/、state.db）覆盖了 SOUL；
3. SOUL.md 实际还是默认内容（用 `cat /opt/hermes/project-a/SOUL.md` 确认）；
4. Gateway 容器没有重新加载（`docker restart hermes-project-a-gateway` 后新开飞书会话）；
5. 旧 Session 已固化 Prompt。

验证与「简单粗暴」解法：

```bash
# 验证：直接在对话里问「请读取当前项目的 SOUL.md 内容」
# 排查信息
cat /opt/hermes/project-a/SOUL.md
ls -l /opt/hermes/project-a/memories
ls -l /opt/hermes/project-a/sessions
docker logs --tail 100 hermes-project-a-gateway

# 粗暴重置
docker stop hermes-project-a-gateway
mv sessions sessions.bak && mv memories memories.bak && mv state.db state.db.bak
docker start hermes-project-a-gateway
```

**结果（第 86 问）**：机器人回答「根据 SOUL.md 的定义，我是服务于 Hipobuy 项目的 AI 研发协作助手」——SOUL.md 已生效，SOUL.md + MEMORY.md + sessions.json 形成闭环。

### 坑 2：SOUL.md 太实诚，把底层信息全暴露（第 88 问）

SOUL.md 里写了「我是 Hermes / 开发方：Nous Research / 底层模型：Claude」，机器人被问身份时如实全盘托出。解决：删掉底层信息，改为身份定义 + 「禁止主动提及 Hermes / Nous Research / Claude / Anthropic，除非用户明确询问底层技术实现」的分层应答规则。

### 坑 3：手册贴完整 AI 输出导致文档过长（第 93/94 问）

真实请求的完整回答直接贴文字，用户「根本不会看」。解决：模板 + 截图 + 摘要式结论，完整输出留在飞书消息里点「展开」查看。

### 坑 4：汇报草稿像技术笔记（第 98/99 问）

> （截图内容根据上下文推断：用户拟发给领导的飞书消息草稿）

问题：`gitlab代码库` 应写 `GitLab 代码仓库`；「记忆、上下文管理」表述重复；`项目背景(SOLU.md)` 领导看不懂（且拼写有误）；附件前缺「建议试点」的引导。解决：用润色后的统一模板（见第六节第 2 小节）。

## 八、要点总结

- SOUL.md 是项目 Agent 的灵魂文件，务必把官方默认 Prompt 替换为项目背景、技术栈、约束、目标、评审原则和输出要求；这是通用 AI 变成项目专属 Agent 的第一步。
- CONTEXT.md / ROADMAP.md / DECISIONS.md 现阶段不必拆分——SOUL.md 超 300 行、项目超 5 个、Agent 超 3 个或开始做企业知识库时再拆。
- SOUL.md 修改后不会实时生效：它通常只在项目/Agent/Session 初始化时加载，state.db 与 memories 会缓存旧记忆，需重启 gateway 并新开会话（必要时备份重置 sessions/memories/state.db）。
- 去 Hermes 化要改四处：飞书应用名称、头像、机器人昵称，以及 SOUL.md 顶部的 System Identity Override + 禁止主动提及底层（Hermes/Nous Research/Claude/Anthropic），并设计分层应答话术。
- SOUL.md 里不要写底层模型和框架来源，否则模型会「太老实」地全部暴露，容易被当成套壳工具。
- 使用手册结构：产品介绍 → 能解决什么问题 → 三分钟快速上手 → 如何使用 → 推荐提问示例 → 注意事项 → FAQ；模板为主 + 3~5 个真实案例截图 + 摘要式结论，不贴几千字完整输出。
- 必备文档四份：用户使用手册、提问规范、管理员手册、能力边界文档（避免背锅）。
- 手册分发：给领导 PDF（正式、格式稳定），给研发 Markdown（可复制 Prompt、可版本管理、放 GitLab Wiki）。
- 汇报口径统一为「企业内部 AI 研发协作平台一期 MVP，申请小范围试点」，不说「AI 聊天机器人」；n8n 经历包装为「快速试错、收敛方案」。
- 效率数字要保守（+50%/+60%/-40%），准确率验证表（AI 结论 vs 人工验证）是最有说服力的一页；主动提出 10 分钟现场演示。
- 汇报时机：临近下班发「1 分钟能看完的简洁版 + PDF + 截图」，正式讨论和演示留到次日上午 9:30~10:30；领导回「ok」后不追问，次日自然跟进约演示。
- 演示案例优选：角色复制（秒级定位代码）、邮政物流刷新限制（风险+边界+工时）、微店 offerID（发现紧急兼容风险，最有杀伤力）。
