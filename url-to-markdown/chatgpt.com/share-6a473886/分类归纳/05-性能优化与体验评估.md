# 05 性能优化与体验评估（第 106~127 问）

## 本篇概述

本篇聚焦 Hermes Agent（对外名 HipobuyAgent）在飞书场景下的响应速度与使用体验优化。核心结论是：真正的耗时大头不是模型推理，也不是授权弹窗或群聊链路，而是 ReAct 模式下多轮 `execute_code` 工具调用（占 50%~70%）。围绕这一根因，对话给出了完整的优化路线：聚合检索脚本 → SOUL.md 项目导航化 → 快速/深度双模式 → 代码索引（SQLite）→ 立即反馈，并讨论了企业响应时间分级标准、系统整体打分、试点汇报话术，以及工时评估拆分为「总工时 vs 交付周期」的 Prompt 改法。

---

## 一、Tool Approval 授权与工具日志暴露（第 106~107 问）

（根据上下文推断：用户截图为飞书群中出现 `execute_code: from hermes_tools import terminal ...` 和 `✅ Approved permanently` 的消息。）

- `Approved permanently` 是 Hermes 的工具授权机制：`execute_code` 可真正执行代码，默认需授权；点过「永久允许」后不再弹窗，**不是报错**。
- 人数分级影响：单人几乎无影响；10 多个研发每人授权一次可接受；100 多人时体验开始下降（大量新用户会问「为什么让我授权」）。
- **更大的问题不是授权，而是工具日志暴露在聊天记录中**：`execute_code...`、`Approved permanently...` 会让产品经理误以为「机器人坏了」，破坏聊天连续性。
- 隐藏方案三档：
  1. 方案一（现状）：Hermes 直接调 `execute_code`，聊天里有 Tool 日志；
  2. 方案二（推荐）：封装 Python API（如 `GET /api/gitlab/pm/98`），Hermes 走 `requests.get()`，飞书只看到「开始分析…/分析完成」；
  3. 方案三（企业最终）：Hermes → 内部 Service → GitLab/Redis/向量库/数据库，聊天过程完全看不到 Tool（大厂 Agent 方式）。
- 结论：这是**体验层优化项，不是架构问题**，不会影响向领导演示，无需为此停下来。企业落地优先级排序：

| 优先级 | 事项 |
| --- | --- |
| 1 | GitLab Webhook 自动触发评审（最高） |
| 2 | 企业飞书应用正式上线 |
| 3 | 完善项目知识（SOUL、Skill、Memory） |
| 4 | 多项目隔离 |
| 5 | 隐藏 execute_code / Tool 日志（体验优化，可放后面） |

此阶段 ChatGPT 给出的整体评分：能力 9.5/10、准确率 9/10、企业落地价值 9/10、飞书交互体验 7.5~8/10（扣分点：Markdown 在飞书展示效果一般 + 工具日志暴露）。

## 二、响应时间可接受标准（分场景分级）

### 用户心理阈值（第 109 问）

```text
10~30秒：体验较好
30~60秒：可接受
1~2分钟：勉强接受
超过2分钟：会觉得慢
```

### 分场景标准（第 114、115、125 问）

- **需求深度评审**：3~5 分钟可接受（人工评审本来要 30 分钟~1 小时，AI 已快很多）；7 分钟偏长、勉强接受。
- **随手问代码/日常群聊**：最好 10~30 秒，等 5~7 分钟不能接受。
- **给领导演示**：不建议用未预热的大需求现场演示，选已跑过、结果稳定、耗时 30~90 秒的案例。

### 企业场景可接受时间对照表（第 125 问）

| 场景 | 用户可接受时间 | 当时状态 |
| --- | --- | --- |
| 普通聊天 | 3~10 秒 | 正常 |
| 查 GitLab 分支、提交 | 10~20 秒 | 正常 |
| 需求评审（代码量小） | 30~60 秒 | 可以接受 |
| 需求评审（中等） | 60~90 秒 | 勉强可以 |
| 大需求、跨模块分析 | 2~3 分钟 | 能接受，但开始影响体验 |
| 3 分钟以上 | 用户开始怀疑卡住了 | 不理想 |
| 5~8 分钟 | 企业基本不能接受 | 需优化 |

### 最终目标体验

```text
首次回复：10~20秒内给"正在分析"
完整摘要：60秒内返回
详细代码分析：用户需要时再展开

普通问题：30~60 秒
需求快速评审：60~90 秒
深度代码评审：3~5 分钟
```

## 三、慢的根因分析

### 耗时拆解（第 108、116、117 问）

- `execute_code` 消息本身：< 10ms，可忽略；`Approved permanently`：仅第一次等待点击（几十到几百毫秒），之后不再耗时。
- **真正耗时的是工具调用次数**：一次评审连续 `grep Controller → grep Service → grep Mapper → cat 文件…`，每次 Tool Call 约 0.5~2 秒；20 次约 20~40 秒，60 次约 2~4 分钟。
- Hermes 是 **ReAct Agent**：每一步都是「LLM 思考 → 工具 → LLM 思考 → 工具」，8 次 `execute_code` 意味着 Claude 推理了 8 次——**Claude 真正输出内容其实很快，慢在反复探索**。
- 问题定性：**70% 在工具层（execute_code / 搜索方式），30% 在 Prompt 层（SOUL.md）**。

### 各环节耗时占比经验值（第 122 问）

| 环节 | 占比 |
| --- | ---: |
| 飞书群聊 vs 私聊 | 1%~3% |
| Memory / Session 加载 | 5%~10% |
| 图片识别 | 10%~20%（仅图片需求） |
| LLM 推理 | 15%~20% |
| **execute_code 多轮工具调用** | **50%~70%** |

### 现状本质

当前 Hermes 更像 Claude Code：边思考边 `grep/cat/find` 自主探索，「每个问题都按深度代码评审在跑」。企业上线应改为：调用聚合接口一次拿回结构化结果（`{controller:[], service:[], mapper:[], commits:[], branches:[]}`），Hermes 直接分析。

## 五、聚合检索脚本 analyze_requirement.py（第 111、118 问）

核心思想：把「grep A → grep B → grep C → cat D → cat E」多次调用，收敛成一次 `python3 /workspace/analyze_requirement.py "<需求关键词>"`，一次返回相关 Controller / Service / Mapper / 前端文件 / 接口 / 最近提交。

脚本要点（第 118 问给出完整实现，放在宿主机 `/srv/gitlab/`，容器内挂载为 `/workspace/`）：

- **业务关键词扩展表**（BIZ_KEYWORDS）：中文业务词 → 英文类名/关键词，如 `订单 → order/Order/OrderItem/导出/export`、`提现 → withdraw/WithdrawApply`、`海关 → customs/declare/申报/大类`；
- 扫描 `aps-boot / admin-page / web-page / h5-page` 四个项目，排除 `.git/node_modules/target/dist` 等目录，限定扩展名；
- **文件分类 + 优先级打分**：按路径含 controller/serviceimpl/service/mapper/enum/task/vue 分类，Controller 权重最高（100），并对 `aps-admin`、`aps-api` 等模块加分；
- 只截取**命中关键词附近的代码片段**（约 12 行），限制文件数（Top 20~35）与片段数，控制返回体积；
- 附带最近 30~60 天相关 `git log` 提交记录；
- 输出末尾内置「给 AI 的分析要求」：按需求结论/代码现状/涉及模块/实现方案/风险点/边界问题/工时评估输出，**证据不足要明确说明，不要强行下结论**。

配套落地步骤：创建脚本 → `chmod +x` → 进容器 `python3 /workspace/analyze_requirement.py "订单导出"` 测试 → 在 SOUL.md 与 pm-review/SKILL.md 中强制优先使用 → `docker restart hermes-project-a-gateway`。

预期效果对比：

```text
优化前：execute_code 10~30 次，耗时 2~5 分钟，飞书里一堆工具日志
优化后：execute_code 1~3 次，耗时 30秒~1分钟，输出更简洁，体验更像企业助手
```

## 六、SOUL.md 优化：从「自我介绍」到「项目导航手册」（第 111~112 问）

SOUL.md 的重点不是「我是谁」，而是 `业务词 → 对应模块 → 对应代码路径`，让 AI 不用每次重新探索目录。第 112 问在用户原有 SOUL.md（含身份定义、禁止暴露 Hermes/Claude 等底层、飞书禁用 Markdown 等）基础上合并出完整版，关键新增结构：

- **项目代码位置**：`/workspace` 下四个项目及职责；
- **后端项目导航**：aps-boot 的 aps-admin / aps-api / aps-common / aps-task / aps-data-support 模块说明，分析顺序 Controller → Service → ServiceImpl → Mapper → XML → Enum → Task；
- **常见业务关键词导航**：订单/物流/支付/提现/角色权限/APP 版本/风控/微店，每类给「关键词 + 优先查找的类 + 常见风险」；
- **工具使用规则**：必须优先执行 `analyze_requirement.py`，禁止一开始大量 grep/find/cat，单次评审控制在 1~3 次工具调用内；
- **高风险模块清单**（订单/支付/提现/物流/定时任务/金额/库存/风控），涉及时必须输出风险等级、回归范围、边界问题、是否灰度、是否需回滚方案；
- **工时规则**：简单 0.5h-2h、普通 0.5d-1d、复杂 2d-5d、高风险 1 周以上，必须结合代码现状判断；
- **飞书输出规范**：禁止 Markdown（`#`、加粗、代码块、表格等），统一用【需求结论】【代码现状】【实现方案】【风险点】【边界问题】【工时评估】【建议方案】+ `━━━━━━━━━━━━━━` 分隔符，默认摘要版，详情引导回复「详细分析」。

Skill（pm-review/SKILL.md）配套加 Tool Efficiency 规则：

```md
# Tool Efficiency Rules

1. 优先执行聚合脚本：python3 /workspace/analyze_requirement.py "<需求关键词>"
2. 禁止一开始连续执行大量 grep、find、cat。
3. 如果聚合脚本结果已包含 Controller/Service/Mapper/前端文件/最近提交，直接基于结果输出评审。
4. 只有证据不足时，才继续读取具体文件。
5. 单次需求评审尽量控制在 1~3 次工具调用内。
6. 默认输出摘要版，不要一次性输出大量代码。
```

## 七、如何检测优化是否生效（第 113 问）

五步验证法：

1. **重启 Hermes**（必须）：`docker restart hermes-project-a-gateway`，等 10~20 秒；
2. **验证 SOUL 重新加载**：飞书问「你是谁」，应回答 HipobuyAgent 而非 Hermes/Claude；
3. **验证是否少探索**：问简单需求，`execute_code` 应只有 1~2 次而不是 ×8/×10/×15；
4. **看工具日志**：`docker logs -f hermes-project-a-gateway`，日志应从几十条 grep/find/cat 收敛为「execute_code → analyze_requirement.py → 返回 → 结束」；
5. **秒表统计响应时间**：记录发送时间与首次开始回复时间（首字时间 TTFB）。

| 指标 | 修改前 | 修改后 |

| 是否大量 grep | 是 | 基本没有 |
| 是否重复探索目录 | 是 | 很少 |

## 八、快速评审 vs 深度评审双模式（第 115、116、119、120 问）

（根据上下文推断：第 115 问两张截图为同一需求 16:14 发起、用户 16:21 追问「还没执行完？」机器人才开始回复，首个有效结果约 7 分钟。）

问题定性：不是「生成答案」慢，而是卡在中间探索阶段，且没有任何中间状态提示，用户不知道它是在工作还是卡住了。解法是拆两个模式：

- **快速评审模式**（日常默认）：只执行 1~2 次代码搜索，30~60 秒内输出摘要（【初步结论】【可能涉及代码】【主要风险】【预计工时】【需要确认】），不输出大段代码；
- **深度评审模式**（用户明确说「深度评审」才触发）：允许跑 3~8 分钟，输出完整代码路径、影响范围、详细方案、风险、测试建议。

SOUL.md 响应速度规则模板：

```md
# 响应速度规则

默认使用"快速评审模式"。

除非用户明确说"深度评审""完整分析""详细代码分析"，否则禁止进行长时间代码探索。

快速评审模式要求：

1. 最多执行 2 次工具调用。
2. 60 秒内必须先给出初步结论。
3. 如果代码证据不足，直接说明"需要深度评审确认"。
4. 不允许为了追求完整性连续搜索多个文件。
5. 不输出大段代码。
6. 回复最后提示："如需完整代码级分析，请回复：深度评审"。
```

第 119 问的关键教训：（根据上下文推断，截图显示加规则后仍 `execute_code ×7`）**「优先」这种措辞对 Agent 不够硬，必须写成「只能执行且仅一次」**：

```md
# 快速评审强制规则

当用户消息包含"快速评审"时，必须进入快速评审模式。

1. 只能执行一次工具调用。
2. 只能执行：python3 /workspace/analyze_requirement.py "<需求关键词>"
3. 禁止再执行 grep、find、cat、ls、sed、awk、python 临时代码。
4. 禁止读取额外文件。
5. 如果一次检索结果不足，直接说明"当前为快速评审，证据不足，建议回复：深度评审"。
```

更彻底的做法是拆两个脚本：`quick_review.py`（只返回 Top 文件 + 关键片段 + 初步模块判断）与 `deep_review.py`（返回更多文件与调用链），在 SOUL.md 中绑定「快速评审只能调用 quick_review.py」。

第 120 问确认见效：17:19 发起、17:21 开始输出（约 2 分钟），工具调用已变为一次 `python3 /workspace/analyze_requiremen...`，比之前 5~7 分钟明显进步；剩余问题是 terminal 日志仍暴露、还需压到 60~90 秒。

优化目标：`快速评审 execute_code ×1，深度评审 execute_code ×3 以内`。

## 九、输入方式的影响：文字 vs 图片（第 121 问）

（根据上下文推断：两张截图对比同类需求，文字提问约 2 分钟返回，纯截图提问近 8 分钟未返回。）

- 文字提问关键词清晰（如「退货订单 审核操作拒绝」），可直接进聚合脚本；
- 图片提问要先做「识别 → 提取需求点 → 转关键词 → 再查代码」的额外链路，且图片字小信息多会反复确认，导致大量 `execute_code`；
- 落地规则：**评审需求必须带文字标题，图片只能作为补充，不能只发图片**；使用手册中加提示「仅上传截图会增加识别与检索时间」。
- SOUL.md 可加图片处理规则：只发图片时先提取标题关键词、默认只做快速评审、图片不清晰时要求补充文字、禁止因图片信息不足连续执行大量工具调用；文字标题与图片同时存在时优先用文字标题检索。

## 十、群聊 vs 私聊差异（第 122 问）

群聊确实比私聊慢，但四个环节加起来不超过 5 秒，绝不是 6 分钟的原因：

1. **飞书消息链路**：群聊多一层 @ 消息解析，约 100~300ms，可忽略；
2. **群上下文**：需判断谁 @、回复谁、是否引用、是否续上一轮，Context 更长，约 0.5~2 秒；
3. **历史 Memory / Session**：试点群几百条消息 + MEMORY.md + sessions.json + state.db，每次读取整理约 3~10 秒——**试点群越聊越慢**；
4. **引用消息（Reply）**：先读 Reply 再读图片再读上下文，进一步增加 Context。

验证方法：同一需求（如「订单导出 请快速评审」）私聊和群聊各测一次记录时间差；差 2~5 秒说明群聊无影响，差 1 分钟则是 Gateway 群消息处理问题。

运营建议：不要一个试点群一直聊半年；按项目建群（需求结束重建）或定期清 Session，否则 Context 越来越大。结论：优化精力不要放在「群聊还是私聊」，而是继续「把工具层做重、让 Agent 做轻」。

## 十一、代码索引（Code Index）思路（第 123~124 问）

### 为什么脚本还不够（第 123 问）

（根据上下文推断：截图为「已确认退货订单增加回退待确认」需求，仍执行 6~7 次 execute_code、耗时约 3 分钟。）

- 根因：**不是脚本慢，而是 Agent 对脚本返回结果「不满意」又自己继续查**——Hermes 是「执行脚本 → LLM 看结果 → 觉得不够 → 继续 execute_code」的循环；
- 排查方法：不要只看次数，要看每次 `execute_code` 具体执行了什么（是脚本之后继续 grep/cat，还是脚本本身多次）；
- 对策一：聚合脚本不能只返回文件名，要返回**文件名 + 关键方法 + 代码片段 + Git Commit + 调用关系**，让 Agent 不用再读文件；
- 对策二：增加**业务语义扩展**——如「回退」自动扩展为 rollback/status/enum/审核/审批/reject/confirm/waitConfirm（状态机类需求单靠字面关键词命中率低）。

### 索引方案设计（第 124 问）

核心思想：**不要让 Agent 每次临时 grep 全仓库，而是提前把代码整理成数据库，评审时只查数据库**。

```text
建索引（后台，cron 每 10 分钟或每天）：
  拉代码（update-all.sh）→ 扫描四个项目 → 解析 Java/注解/XML SQL/Vue 接口调用/git log
  → 写入 /srv/gitlab/.hipobuy-code-index/code_index.db（SQLite）

查询（前台，一次 execute_code）：
  python3 /workspace/query_code_index.py "退货订单 审核拒绝 回退待确认"
  → 返回【命中模块】【相关接口】【相关代码及行号】【关键片段】【最近提交】【建议继续确认】
```

v1 只做 5 张表，不要做复杂：

| 表 | 主要字段 |
| --- | --- |
| files | id, project, path, ext, mtime, file_type, content_hash |
| symbols | file_id, symbol_type(class/method/enum/…), symbol_name, class_name, start_line, end_line, annotations, snippet, keywords |
| api_routes | project, file_path, class_name, method_name, http_method, route, start/end_line, snippet |
| snippets | file_id, keyword, start_line, end_line, snippet |
| git_commits | project, hash, author, date, message, files |

配套 `biz_keywords.yaml` 维护中文业务词映射（退货→return/ReturnOrder/aftersale，提现→withdraw/WithdrawApply 等），查询时自动扩展。与 `analyze_requirement.py` 的区别：前者每次现场扫描，索引是**提前扫好、现场只查询**（10ms 级 SQL 而非 10 秒级 grep），预计从 3~8 分钟降到 30~90 秒。第一版明确**不上向量库、不上 MCP**。

## 十二、立即反馈：心理体验优化（第 125 问）

Claude Code 感觉快的真相：它总时间可能也是 2 分钟，但会持续输出 `Thinking... / Reading... / Searching...`，让用户感觉一直在工作——**心理体验完全不同**。

当前系统最缺的不是「少一分钟」，而是**立即反馈**：

```text
收到消息立刻：收到需求，开始分析代码……预计需要 1~2 分钟
20 秒后：正在分析：✔ Controller ✔ Service，正在读取 Mapper……
最后：评审完成 + 结果
```

即使总时间不变，体验会提升一倍。群聊场景尤其重要：AI 3 分钟不说话，消息被刷走、上下文断裂，是群聊最大的体验问题。

## 十四、工时评估双维度：总工时 vs 交付周期（第 127 问）

（根据上下文推断：截图为领导反馈，建议工时评估拆成「一个总耗时、一个实际耗时」。）

领导建议合理且应采纳，本质是区分两个概念（腾讯/阿里/字节通用做法）：

- **总工时（Person Hour）**：所有参与人员投入时间累计 → 评估人力成本、资源投入；
- **交付周期（Elapsed Time）**：考虑前后端能否并行后的实际完成时间 → 评估上线时间、排期。

示例：前端 1h + 后端 1h + 测试 1h——串行时总工时 3h、周期 3h；前后端并行时总工时 3h、周期 2h；多人并行甚至可能总工时 7h、交付 3h。

Prompt 增加规则：

```text
工时评估时，必须区分：

① 总工时（所有参与人员投入时间累计）
② 实际交付周期（考虑前后端是否可以并行开发后的完成时间）

若无法判断是否可并行，请明确说明假设条件。
```

推荐的企业级统一输出格式：

```text
【工时评估】

人员投入（总工时）
后端：2h
前端：1h
测试：1h
合计：4人小时

━━━━━━━━━━━━━━

预计交付周期
前后端可并行：约3小时
若串行开发：约4小时

━━━━━━━━━━━━━━

说明
以上周期基于：
1、需求确认完成；
2、不包含测试反馈后的返工；
3、无其它高优先级任务插入。
```

此规则建议写入 SOUL.md。

## 踩坑与解决

| 坑 | 现象 | 解决 |
| --- | --- | --- |
| SOUL 引用了不存在的脚本 | SOUL.md 写了「优先执行 analyze_requirement.py」但脚本没创建，AI 找不到就退回 grep/find/cat 循环 | 先把脚本真正做出来，再写规则；改 SOUL 前先确认工具可用 |
| 「优先」措辞约束不住 Agent | 加了规则后仍 `execute_code ×7` | 把「优先执行」改成「只能执行且仅一次」，并显式列出禁止的命令（grep/find/cat/ls/sed/awk/临时 python） |
| 脚本只返回文件名 | Agent 觉得「还不知道里面有没有 checkSave()」，继续 cat 文件 | 脚本返回关键方法 + 代码片段 + 行号 + Git 提交，一次给足 90% 信息 |
| 字面关键词命中率低 | 状态机类需求（「回退待确认」）搜不到相关代码，AI 连续 grep | 增加业务语义扩展（回退→rollback/status/enum/审核/reject/confirm 等），维护 biz_keywords.yaml |
| 只发截图评审 | 图片需求近 8 分钟未返回，文字需求 2 分钟返回 | 规定需求必须带文字标题，图片仅作补充；SOUL.md 加图片处理规则 |
| 试点群越聊越慢 | 群历史几百条消息 + Memory/Session 全量加载 | 按项目建群、需求结束重建群，或定期清 Session |
| 无中间反馈被误认为卡死 | 3 分钟无响应，用户重新 @、重发甚至关聊天 | 收到消息立即回「开始分析，预计 X 分钟」，分阶段输出进度 |
| 反复打磨 Prompt 收益递减 | SOUL.md 已较完善后继续改，收益越来越小 | 精力转向代码检索能力（聚合脚本 → 代码索引 → GitLab MCP） |

## 要点总结

- 耗时大头是多轮 `execute_code` 工具调用（占 50%~70%），不是模型推理、不是授权弹窗、不是群聊链路；`Approved permanently` 只在首次授权耗时。
- 工具日志暴露（execute_code / Approved permanently）是体验问题而非架构问题，可通过封装 API / 内部 Service 逐步隐藏，不必为此停下试点。
- 响应时间标准分场景：普通问答 3~10 秒、查代码 20~40 秒、需求评审 40~90 秒；超过 3 分钟用户开始怀疑卡死，5~8 分钟企业基本不能接受。
- 优化路线按投入产出比：聚合脚本（一次 execute_code 拿回全部证据）→ SOUL.md 项目导航化（业务词→模块→代码路径）→ 快速/深度双模式 → SQLite 代码索引（提前扫描、现场只查）→ 后期再考虑 RAG/向量库。
- 对 Agent 的规则要写「硬」：用「只能执行且仅一次」而非「优先执行」，并显式禁用零散 grep/find/cat；聚合脚本必须返回方法级片段，否则 Agent 会继续自主探索。
- 立即反馈（收到即回「开始分析」+ 分阶段输出）在总耗时不变的情况下能把心理体验提升一倍，是成本最低的体验优化。
- 输入规范影响巨大：文字标题必带、截图仅作补充；群聊比私聊多的开销不超过 5 秒，但试点群要防 Context 膨胀（定期清 Session / 按项目建群）。
- 系统当前定位 V1 内部试点版：功能能力和评审质量已达试点条件，最大短板是响应速度，目标平均 1 分钟内、复杂需求 2 分钟内。
- 工时评估应拆双维度输出：总工时（Person Hour，人力成本）+ 实际交付周期（Elapsed Time，排期），并说明并行/串行假设条件，该规则应写入 SOUL.md。
