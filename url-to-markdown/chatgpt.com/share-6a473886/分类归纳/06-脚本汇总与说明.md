## 一、部署与运行：Docker 命令与配置

### 1.1 Setup 向导（初始化配置）（Q9~Q11、Q20、Q22）

**用途**：首次初始化 Hermes Agent 配置（模型、协议、消息通道等），配置写入挂载的数据目录。`--rm` 表示容器用完即删，配置因挂载而持久保留——这是一次性命令，不是常驻服务。

```bash
# 单实例（个人测试）
docker run -it --rm \
  -v ~/.hermes:/opt/data \
  nousresearch/hermes-agent setup

# 企业多项目：每个项目一个独立数据目录
docker run -it --rm \
  --name hermes-project-a-setup \
  -v /opt/hermes/project-a:/opt/data \
  nousresearch/hermes-agent setup
```

**子命令变体**（只重跑某一部分配置，不必全量重来）：

```bash
# 只重新配置模型
docker run -it --rm -v /opt/hermes/project-a:/opt/data \
  nousresearch/hermes-agent setup model

# 只配置 gateway（消息通道，如飞书）
docker run -it --rm -v /opt/hermes/project-a:/opt/data \
  nousresearch/hermes-agent setup gateway

# 只配置工具
docker run -it --rm -v /opt/hermes/project-a:/opt/data \
  nousresearch/hermes-agent setup tools
```

**说明**：BYOK（自带 Key）场景下，向导中选择 Full setup + Custom endpoint，密钥与中转站地址写入数据目录下的 `.env`，即可完全绕开 Nous Portal 官方授权。

### 1.2 BYOK 环境变量配置（Q2~Q7、Q64、Q78）

**用途**：让 Hermes 使用自有 API/中转站而非 Nous 订阅。写入各项目数据目录（容器内 `/opt/data/.env`，宿主机对应 `/opt/hermes/project-a/.env`）。

```env
# 模型（中转站为 OpenAI-compatible 协议）
OPENAI_API_KEY=你的中转站Key
OPENAI_BASE_URL=https://api.zetaapi.ai/v1
ANTHROPIC_API_KEY=

# GitLab（供 gitlab-api.sh 读取）
GITLAB_BASE_URL=https://tool-gitlab.therow-online.com
GITLAB_TOKEN=你的只读PAT

# 飞书（供 gateway 通道使用）
FEISHU_APP_ID=cli_xxx
FEISHU_APP_SECRET=xxx
```

**说明**：对话中用户的中转站是 `https://api.zetaapi.ai`，协议按 OpenAI-compatible（Auto-detect）处理；换飞书机器人（换 AppID）只需改这两个 FEISHU 变量后重启容器。

### 1.3 Gateway 常驻服务（最终定稿形态）（Q27~Q29、Q35、Q40、Q43）

**用途**：后台常驻运行，承接飞书等消息通道。这是整个部署的最关键命令。

```bash
docker run -d \
  --name hermes-project-a-gateway \
  --restart unless-stopped \
  -v /opt/hermes/project-a:/opt/data \
  -v /srv/gitlab:/workspace \
  nousresearch/hermes-agent hermes gateway
```

**关键点**：

- **必须显式写 `hermes gateway` 子命令**。对话中的核心踩坑：`docker run -d` 跑默认命令时，容器因无交互终端而无限重启（Q23~Q27），日志正常但服务起不来；显式指定 `hermes gateway` 后才真正常驻。
- `-v /srv/gitlab:/workspace`：把宿主机代码仓库目录挂进容器，Agent 才能读到 GitLab 代码（Q33 之后加入）。加挂载需要先 `docker rm -f hermes-project-a-gateway` 再重建容器。
- gateway 模式的行为（Q28）：一直在后台运行，人在不在服务器都不影响；想交互时另开临时聊天容器（见 1.4），二者共享同一数据目录，记忆互通。

### 1.4 交互聊天 / 调试容器（Q26~Q29、Q33）

**用途**：临时进入交互式对话（或 bash 调试），与 gateway 共享同一数据目录，用完即删。

```bash
# 交互聊天（与 gateway 共存，不冲突）
docker run -it --rm \
  --name hermes-project-a-chat \
  -v /opt/hermes/project-a:/opt/data \
  -v /srv/gitlab:/workspace \
  nousresearch/hermes-agent hermes

# 进容器 bash 排查（看 .env、目录结构等）
docker run -it --rm \
  --name hermes-project-a-debug \
  -v /opt/hermes/project-a:/opt/data \
  nousresearch/hermes-agent bash

# 或直接进入正在运行的 gateway 容器
docker exec -it hermes-project-a-gateway bash
cat /opt/data/.env
env | grep FEISHU
```

### 1.5 多实例隔离：docker compose（Q11）

**用途**：多个项目各跑一个 Hermes 实例，配置、记忆、密钥完全隔离（一个项目 = 一个容器 + 一个独立数据目录）。

```yaml
services:
  hermes-project-a:
    image: nousresearch/hermes-agent
    container_name: hermes-project-a
    restart: unless-stopped
    volumes:
      - /opt/hermes/project-a:/opt/data

  hermes-project-b:
    image: nousresearch/hermes-agent
    container_name: hermes-project-b
    restart: unless-stopped
    volumes:
      - /opt/hermes/project-b:/opt/data
```

**说明**：此为对话早期版本；按 1.3 的结论，command 实际也应显式指定 `hermes gateway`，并按需追加 `/workspace` 挂载。目录规划（Q39）：

```text

├── project-a
└── project-b

/srv/gitlab          # 代码仓库（与 Agent 数据分离）
├── aps-boot
├── admin-page
```

**用途**：整套系统的可迁移备份——Agent 数据目录 + 代码目录打包即可在新机器恢复。

## 二、GitLab 代码同步脚本

### 2.1 初次克隆（Q33、Q34、Q36）

**用途**：把 GitLab 仓库以只读 Token 克隆到宿主机 `/srv/gitlab`，作为 Agent 的代码分析主数据源。

```bash
mkdir -p /srv/gitlab
cd /srv/gitlab

# --depth 1 浅克隆：只拉最新版本，磁盘占用最小（10 个以内仓库足够）
git clone --depth 1 https://oauth2:<你的PAT>@tool-gitlab.therow-online.com/group/project-a.git
git clone --depth 1 https://oauth2:<你的PAT>@tool-gitlab.therow-online.com/group/project-b.git
```

**说明**：`oauth2:<TOKEN>@` 是 GitLab PAT 走 HTTPS 克隆的标准写法；Token 只需 `read_repository`/`read_api` 权限。

### 2.2 全仓库更新脚本 update-all.sh（Q38、Q40）

**用途**：解决「GitLab 每天都有人提交合并，本地代码如何保持最新」的问题。遍历 `/srv/gitlab` 下所有 git 仓库统一拉取。

**存放路径**：`/srv/gitlab/update-all.sh`（`chmod +x` 赋执行权限）

```bash
#!/bin/bash
set -e

BASE_DIR="/srv/gitlab"

for repo in "$BASE_DIR"/*; do
  if [ -d "$repo/.git" ]; then
    echo "Updating: $repo"
    cd "$repo"
    git fetch --all --prune
    git pull --ff-only
    echo "Done: $repo"
    echo "----------------"
  fi
done
```

**要点说明**：

- `git pull --ff-only`：只做快进合并，本地目录是纯只读镜像、绝不产生合并提交，避免任何冲突。
- `git fetch --all --prune`：同步所有远端分支并清理已删除分支。
- 只认含 `.git` 的子目录，脚本文件本身不受影响。

### 2.3 crontab 定时同步（Q38）

**用途**：每 10 分钟自动执行同步，Agent 分析的永远是准新鲜代码。

```bash
crontab -e
# 追加：
*/10 * * * * /srv/gitlab/update-all.sh >> /var/log/gitlab-repo-sync.log 2>&1
```

### 3.2 gitlab-api.sh 只读封装脚本（Q43）

**用途**：把常用 GitLab 只读 API 封装成一个脚本，Agent 在容器内一条命令即可查项目、分支、MR、提交（元数据类信息走 API，代码内容走本地 clone——这是对话确定的分工）。

**存放路径**：`/srv/gitlab/gitlab-api.sh`（容器内路径 `/workspace/gitlab-api.sh`）

```bash
#!/bin/bash
set -e

ENV_FILE="/opt/data/.env"
[ -f "$ENV_FILE" ] && source "$ENV_FILE"

BASE="${GITLAB_BASE_URL%/}"
TOKEN="$GITLAB_TOKEN"

if [ -z "$BASE" ] || [ -z "$TOKEN" ]; then
  echo "Missing GITLAB_BASE_URL or GITLAB_TOKEN"
  exit 1
fi

case "$1" in
  me)
    curl -s --header "PRIVATE-TOKEN: $TOKEN" "$BASE/api/v4/user"
    ;;
  projects)
    curl -s --header "PRIVATE-TOKEN: $TOKEN" "$BASE/api/v4/projects?simple=true&per_page=100"
    ;;
  branches)
    curl -s --header "PRIVATE-TOKEN: $TOKEN" "$BASE/api/v4/projects/$2/repository/branches"
    ;;
  mrs)
    curl -s --header "PRIVATE-TOKEN: $TOKEN" "$BASE/api/v4/projects/$2/merge_requests?state=opened"
    ;;
  commits)
    curl -s --header "PRIVATE-TOKEN: $TOKEN" "$BASE/api/v4/projects/$2/repository/commits?per_page=20"
    ;;
  *)
    echo "Usage:"
    echo "$0 me"
    echo "$0 projects"
    echo "$0 branches <project_id>"
    echo "$0 mrs <project_id>"
    echo "$0 commits <project_id>"
    ;;
esac
```

**使用方式**（容器内，Q78 使用手册中的标准用法）：

```bash
bash /workspace/gitlab-api.sh projects       # 列出全部项目（拿 project_id）
bash /workspace/gitlab-api.sh commits 7      # 项目 7 最近 20 条提交
bash /workspace/gitlab-api.sh branches 7     # 项目 7 分支列表
bash /workspace/gitlab-api.sh mrs 7          # 项目 7 打开中的 MR
```

**要点说明**：

- 密钥不写死在脚本里，从 `/opt/data/.env` 读取（`source`），脚本可以放心进 git。
- `${GITLAB_BASE_URL%/}` 去掉末尾斜杠，避免拼出 `//api/v4`。
- 只封装了只读端点，与只读 Token 的权限边界一致。

### 3.3 容器内外路径兼容片段（Q47）

**用途**：踩坑修复——同一脚本在宿主机执行时读不到容器路径 `/opt/data/.env`。用路径探测做 fallback，让脚本在容器内外都能跑。

```bash
if [ -f "/opt/data/.env" ]; then
  ENV_FILE="/opt/data/.env"                # 容器内
elif [ -f "/opt/hermes/project-a/.env" ]; then
  ENV_FILE="/opt/hermes/project-a/.env"    # 宿主机
else
  ENV_FILE=""
fi

[ -n "$ENV_FILE" ] && source "$ENV_FILE"
```

## 四、GitLab Webhook 接收服务

### 4.1 server.py：标准库版 Webhook 接收器（Q54）

**用途**：接收 GitLab 的 Merge Request 事件（创建 / 更新 / 合并），校验 Secret Token 后：① 把事件 JSON 落盘存档；② MR 打开/更新时 fetch 对应分支、MR 合并时触发全量 `update-all.sh` 同步。零第三方依赖（纯 Python 标准库）。

**存放路径**：`/opt/gitlab-webhook/server.py`

```python
#!/usr/bin/env python3
import json
import os
import subprocess
from datetime import datetime
from http.server import BaseHTTPRequestHandler, HTTPServer

SECRET = os.environ.get("GITLAB_WEBHOOK_SECRET", "")
PORT = int(os.environ.get("GITLAB_WEBHOOK_PORT", "9099"))

REPO_BASE = "/srv/gitlab"
EVENT_DIR = "/srv/gitlab/webhook-events"
UPDATE_SCRIPT = "/srv/gitlab/update-all.sh"


def run_cmd(cmd):
    try:
        result = subprocess.run(
            cmd,
            shell=True,
            text=True,
            capture_output=True,
            timeout=120,
        )
        return {
            "cmd": cmd,
            "returncode": result.returncode,
            "stdout": result.stdout[-4000:],
            "stderr": result.stderr[-4000:],
        }
    except Exception as e:
        return {
            "cmd": cmd,
            "error": str(e),
        }


def safe_write_event(data):
    os.makedirs(EVENT_DIR, exist_ok=True)
    ts = datetime.now().strftime("%Y%m%d_%H%M%S")
    project = data.get("project", {}).get("name", "unknown")
    action = data.get("object_attributes", {}).get("action", "unknown")
    iid = data.get("object_attributes", {}).get("iid", "unknown")

    filename = f"{ts}_{project}_mr{iid}_{action}.json"
    path = os.path.join(EVENT_DIR, filename)

    with open(path, "w", encoding="utf-8") as f:
        json.dump(data, f, ensure_ascii=False, indent=2)

    return path


def handle_merge_request(data):
    obj = data.get("object_attributes", {})
    project = data.get("project", {})

    action = obj.get("action")
    state = obj.get("state")
    title = obj.get("title")
    iid = obj.get("iid")
    source_branch = obj.get("source_branch")
    target_branch = obj.get("target_branch")

    project_name = project.get("name")
    repo_path = os.path.join(REPO_BASE, project_name) if project_name else ""

    summary = {
        "event": "merge_request",
        "action": action,
        "state": state,
        "project": project_name,
        "mr_iid": iid,
        "title": title,
        "source_branch": source_branch,
        "target_branch": target_branch,
        "commands": [],
    }

    if repo_path and os.path.isdir(os.path.join(repo_path, ".git")):
        if action in ["open", "reopen", "update"]:
            summary["commands"].append(
                run_cmd(
                    f"git -C {repo_path} fetch origin "
                    f"{source_branch} {target_branch} --prune"
                )
            )

        if action == "merge" or state == "merged":
            summary["commands"].append(run_cmd(UPDATE_SCRIPT))
    else:
        summary["commands"].append({
            "warning": f"Local repo not found: {repo_path}"
        })

    return summary


class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        if self.path == "/health":
            self.send_response(200)
            self.end_headers()
            self.wfile.write(b"ok")
            return

        self.send_response(404)
        self.end_headers()

    def do_POST(self):
        if self.path != "/gitlab/webhook":
            self.send_response(404)
            self.end_headers()
            return

        token = self.headers.get("X-Gitlab-Token", "")
        if SECRET and token != SECRET:
            self.send_response(403)
            self.end_headers()
            self.wfile.write(b"invalid token")
            return

        length = int(self.headers.get("Content-Length", 0))
        raw = self.rfile.read(length)

        try:
            data = json.loads(raw.decode("utf-8"))
        except Exception:
            self.send_response(400)
            self.end_headers()
            self.wfile.write(b"invalid json")
            return

        event_type = self.headers.get("X-Gitlab-Event", "")
        event_file = safe_write_event(data)

        result = {
            "received": True,
            "event_type": event_type,
            "event_file": event_file,
        }

        if event_type == "Merge Request Hook":
            result["result"] = handle_merge_request(data)
        else:
            result["ignored"] = "not merge request event"

        print(json.dumps(result, ensure_ascii=False, indent=2))

        self.send_response(200)
        self.end_headers()
        self.wfile.write(json.dumps(result, ensure_ascii=False).encode("utf-8"))


if __name__ == "__main__":
    server = HTTPServer(("0.0.0.0", PORT), Handler)
    print(f"GitLab webhook server listening on 0.0.0.0:{PORT}")
    server.serve_forever()
```

**要点说明**：

- `X-Gitlab-Token` 头与环境变量 `GITLAB_WEBHOOK_SECRET` 比对，防伪造请求。
- 每个事件落盘为 `时间戳_项目_mrIID_动作.json`，留存审计与后续 AI 评审的输入。
- `GET /health` 健康检查端点，方便探活。
- stdout/stderr 各截断保留最后 4000 字符，防止大输出撑爆响应。
- **架构结论（Q55）**：Hermes CLI 不支持非交互 `--prompt` 一次性执行，所以 Webhook 自动链路不经过 Hermes——人工分析走 Hermes（飞书对话），自动评审走「Webhook → Python → LLM API」双链路并行。

### 4.2 systemd 服务单元（Q54）

**用途**：让 webhook 服务开机自启、崩溃自动拉起。

**存放路径**：`/etc/systemd/system/gitlab-webhook.service`

```ini
[Unit]
Description=GitLab Webhook Receiver
After=network.target

[Service]
Type=simple
Environment=GITLAB_WEBHOOK_SECRET=你自己设置一个随机密钥
Environment=GITLAB_WEBHOOK_PORT=9099
ExecStart=/usr/bin/python3 /opt/gitlab-webhook/server.py
Restart=always
RestartSec=3

[Install]
WantedBy=multi-user.target
```

**启用命令**：

```bash
systemctl daemon-reload
systemctl enable gitlab-webhook
systemctl start gitlab-webhook
systemctl status gitlab-webhook
```

**GitLab 侧配置**：仓库 Settings → Webhooks，URL 填 `http://<服务器>:9099/gitlab/webhook`，Secret Token 填同一密钥，勾选 Merge request events。

```python
# 第三步：调用 GitLab API 拉取 MR 的 diff（评审输入）
import requests
```

## 七、聚合代码检索脚本 analyze_requirement.py

**背景（Q110~Q118）**：性能诊断发现响应慢的根因是 ReAct 模式下多轮 `execute_code`（grep → find → cat → grep → cat…，占总耗时 50%~70%）。解法是把「搜文件 + 提片段 + 查 git log」聚合成**一个脚本一次执行**，Agent 只调一次工具就拿到全部代码证据。

**演进**：Q111 先给出 v1（约 159 行）；用户反馈「执行工具次数还是多」（Q118）后升级为 v2——增加文件评分排序、目录/扩展名过滤、命中统计和固定输出骨架。v1 已被 v2 完全取代，此处收录 v2 全文。

**存放路径**：`/workspace/analyze_requirement.py`（即宿主机 `/srv/gitlab/analyze_requirement.py`）

**使用方式**：

```bash
python3 /workspace/analyze_requirement.py "退货订单 审核拒绝 回退待确认"
```

**配套 SOUL.md 规则**（Q119 修正：必须写死，不能写「优先」，否则模型仍会自由发挥多次调用）：

```text
代码检索只能执行 python3 /workspace/analyze_requirement.py "<关键词>"，
且整个评审过程只允许执行一次。
```

```python
#!/usr/bin/env python3
import os
import re
import sys
import json
import subprocess
from pathlib import Path
from collections import defaultdict

WORKSPACE = Path("/workspace")

PROJECTS = [
    "aps-boot",
    "admin-page",
    "web-page",
    "h5-page",
]

IGNORE_DIRS = {
    ".git", "node_modules", "target", "dist", "build",
    ".idea", ".vscode", "logs", "cache"
}

ALLOW_EXT = {
    ".java", ".xml", ".yml", ".yaml", ".properties",
    ".vue", ".js", ".ts", ".json"
}

BIZ_KEYWORDS = {
    "订单": ["订单", "order", "Order", "OrderItem", "SubOrder", "子订单", "导出", "export", "Export"],
    "物流": ["物流", "logistics", "Logistics", "shipment", "Shipment", "trace", "Trace", "轨迹"],
    "支付": ["支付", "pay", "Pay", "payment", "Payment", "onlypay", "OnlyPay"],
    "提现": ["提现", "withdraw", "Withdraw", "WithdrawApply"],
    "角色": ["角色", "role", "Role", "permission", "Permission", "菜单", "权限"],
    "版本": ["版本", "version", "Version", "AppVersion", "appVersion"],
    "风控": ["风控", "risk", "Risk", "RiskType", "WeTech"],
    "微店": ["微店", "weidian", "WeiDian", "itemID", "offerID"],
    "海关": ["海关", "customs", "Customs", "declare", "Declare", "申报", "大类", "category"],
    "导出": ["导出", "export", "Export", "excel", "Excel", "download", "Download"],
}

MAX_FILES_TOTAL = 35
MAX_SNIPPETS = 12
MAX_SNIPPET_LINES = 12
MAX_FILE_SIZE = 1024 * 800


def run(cmd, cwd=None, timeout=8):
    try:
        r = subprocess.run(
            cmd,
            shell=True,
            cwd=cwd,
            text=True,
            capture_output=True,
            timeout=timeout,
        )
        return r.stdout.strip()
    except Exception as e:
        return f"[ERROR] {e}"


def expand_keywords(query: str):
    kws = set()

    for p in re.split(r"[\s,，。:：;；/、\-_]+", query):
        p = p.strip()
        if p:
            kws.add(p)

    for k, vals in BIZ_KEYWORDS.items():
        if k in query:
            kws.update(vals)

    return list(dict.fromkeys([x for x in kws if x]))


def should_skip(path: Path):
    return any(part in IGNORE_DIRS for part in path.parts)


def classify(path: str):
    p = path.lower()

    if "controller" in p:
        return "Controller / 接口入口"
    if "serviceimpl" in p:
        return "ServiceImpl / 业务实现"
    if "service" in p:
        return "Service / 业务接口"
    if "mapper" in p and p.endswith(".xml"):
        return "Mapper XML / SQL"
    if "mapper" in p:
        return "Mapper / DAO"
    if "enum" in p:
        return "Enum / 枚举"
    if "task" in p or "scheduled" in p or "job" in p:
        return "Task / 定时任务"
    if p.endswith(".vue"):
        return "Vue / 前端页面"
    if p.endswith(".js") or p.endswith(".ts"):
        return "JS/TS / 前端逻辑"
    if p.endswith(".xml"):
        return "XML / 配置或SQL"
    return "Other"


def priority(path: str):
    c = classify(path)
    score = 0

    weights = {
        "Controller": 100,
        "ServiceImpl": 95,
        "Service": 90,
        "Mapper XML": 80,
        "Mapper": 75,
        "Enum": 65,
        "Task": 60,
        "Vue": 55,
        "JS/TS": 45,
    }

    for k, v in weights.items():
        if k in c:
            score += v

    if "/aps-admin/" in path:
        score += 10
    if "/aps-api/" in path:
        score += 10
    if "/aps-common/" in path:
        score += 5
    if "/admin-page/" in path:
        score += 5

    return score


def read_text(path: Path):
    try:
        if path.stat().st_size > MAX_FILE_SIZE:
            return ""
        return path.read_text(errors="ignore")
    except Exception:
        return ""


def scan_files(keywords):
    hits = defaultdict(lambda: {"score": 0, "matches": defaultdict(int)})

    for project in PROJECTS:
        root = WORKSPACE / project
        if not root.exists():
            continue

        for path in root.rglob("*"):
            if not path.is_file():
                continue
            if should_skip(path):
                continue
            if path.suffix not in ALLOW_EXT:
                continue

            text = read_text(path)
            if not text:
                continue

            rel = str(path)
            base_score = priority(rel)

            for kw in keywords:
                if not kw:
                    continue
                count = text.count(kw)
                if count:
                    hits[rel]["score"] += count * 10 + base_score
                    hits[rel]["matches"][kw] += count

    ranked = sorted(
        hits.items(),
        key=lambda x: x[1]["score"],
        reverse=True,
    )

    return ranked[:MAX_FILES_TOTAL]


def snippets_for_file(path_str, keywords):
    path = Path(path_str)
    text = read_text(path)
    if not text:
        return []

    lines = text.splitlines()
    indexes = []

    for i, line in enumerate(lines):
        if any(kw in line for kw in keywords):
            indexes.append(i)

    blocks = []
    used = set()

    for idx in indexes[:3]:
        start = max(0, idx - 4)
        end = min(len(lines), idx + MAX_SNIPPET_LINES)

        key = (start, end)
        if key in used:
            continue
        used.add(key)

        block = []
        for n in range(start, end):
            block.append(f"{n+1}: {lines[n]}")
        blocks.append("\n".join(block))

    return blocks


def git_commits(project, keywords):
    root = WORKSPACE / project
    if not (root / ".git").exists():
        return ""

    grep_expr = "|".join([re.escape(k) for k in keywords[:8]])
    if not grep_expr:
        return ""

    cmd = (
        "git log --since='60 days ago' "
        "--pretty=format:'%h | %ad | %an | %s' "
        "--date=short --all "
        f"--regexp-ignore-case --grep='{grep_expr}' -n 15"
    )

    return run(cmd, cwd=root, timeout=8)


def print_report(query, keywords, ranked):
    print("【代码检索聚合结果】")
    print()
    print(f"需求关键词：{query}")
    print(f"扩展关键词：{', '.join(keywords)}")
    print()

    print("【相关文件 Top】")
    if not ranked:
        print("未找到明显相关文件。")
    else:
        for path, info in ranked[:20]:
            match_text = ", ".join([f"{k}:{v}" for k, v in list(info["matches"].items())[:8]])
            print(f"- {classify(path)} | {path} | 命中：{match_text}")

    print()
    print("【关键代码片段】")
    for path, info in ranked[:MAX_SNIPPETS]:
        print()
        print(f"文件：{path}")
        print(f"类型：{classify(path)}")
        blocks = snippets_for_file(path, keywords)
        if not blocks:
            print("未截取到有效片段。")
            continue

        for b in blocks[:2]:
            print("片段：")
            print(b)
            print("----")

    print()
    print("【最近相关 Git 提交】")
    for project in PROJECTS:
        commits = git_commits(project, keywords)
        if commits:
            print()
            print(f"项目：{project}")
            print(commits)

    print()
    print("【请基于以上信息输出】")
    print("1. 需求结论")
    print("2. 代码现状")
    print("3. 实现方案")
    print("4. 风险点")
    print("5. 边界问题")
    print("6. 工时评估")
    print("7. 建议方案")
    print()
    print("要求：如果证据不足，明确说明需要进一步确认，不要强行下结论。")


def main():
    query = " ".join(sys.argv[1:]).strip()
    if not query:
        print("用法：python3 /workspace/analyze_requirement.py '<需求关键词>'")
        sys.exit(1)

    keywords = expand_keywords(query)
    ranked = scan_files(keywords)
    print_report(query, keywords, ranked)


if __name__ == "__main__":
    main()
```

**设计要点说明**：

- **关键词扩展**：中文业务词映射到代码中的英文命名（`BIZ_KEYWORDS`），如「订单」自动扩展出 `order/Order/OrderItem/SubOrder` 等，弥合业务语言与代码命名的鸿沟。此映射表需按自己项目的领域词持续维护。
- **文件评分排序**：命中次数 ×10 + 文件类型权重（Controller 100 > ServiceImpl 95 > Service 90 > Mapper XML 80 …），保证接口入口和业务实现排最前。
- **降噪与限流**：跳过 `.git/node_modules/target` 等目录、只扫白名单扩展名、单文件超 800KB 不读、最多 35 个文件 / 12 组片段，输出可控不撑爆上下文。
- **片段带行号**：`行号: 代码` 格式，方便 AI 在评审结论中精确引用位置。
- **git log 联查**：近 60 天、按关键词 grep 提交信息，直接给出「最近谁动过这块」。
- **报告尾部内置指令**：输出末尾直接携带「请基于以上信息输出 1~7」的评审骨架和「证据不足要明说」的约束——脚本输出本身就是喂给模型的结构化 Prompt。

## 十、使用注意事项

- **密钥安全**：对话中用户曾把带 Token 的截图直接发出（Q35 附近），被提醒 Token 已泄露应立即在 GitLab 上 revoke 重发。所有脚本都遵循「密钥进 `.env`、脚本不含密钥」的原则。
- **只读边界**：GitLab Token 仅 read 权限，`gitlab-api.sh` 只封装只读端点，`update-all.sh` 用 `--ff-only` 保证本地仓库纯镜像——整套链路对 GitLab 无任何写操作。MR 评论回写（第五节）因需要写权限被有意暂缓。
- **教学示例定位**：Webhook 服务中 `subprocess.run(shell=True)` 拼接命令、Flask 骨架未做异常兜底等，生产使用前应加固（参数化命令、请求签名重放防护、限流）。
- **哪些是「定稿」**：`update-all.sh`、crontab、`gitlab-api.sh`、gateway docker run（1.3）、`analyze_requirement.py` v2、Skill 目录结构——这些是对话结束时实际在用的版本；Flask 评审片段（第五节）与代码索引（第八节）是规划/半成品。
